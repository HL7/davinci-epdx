# ============================================================================
# mTLS Endpoint X.509 CA certificate signature format (validator limitation)
# ============================================================================
# Reason for suppression: PDex's mTLS Endpoint example uses
# `application/x-x509-ca-cert` as the MIME type identifying the X.509 CA
# certificate exchanged during mTLS trust establishment. The FHIR validator's
# signature-format check has built-in knowledge of only two signature
# formats — `application/jose` (JWS / JWT) and `application/pkcs7-signature`
# — and emits an information-level note when it encounters any other
# signature-format value. `application/x-x509-ca-cert` is the IANA-registered
# media type for X.509 CA certificates and is correct for the mTLS trust
# anchor exchange described in the mTLS Endpoint profile. The validator's
# inability to verify the format is a tooling limitation, not a profile
# defect; cert validation is performed by TLS infrastructure outside FHIR.
%The signature format 'application/x-x509-ca-cert' is not checked (known formats: application/jose and application/pkcs7-signature)%

# canonical URLs (profiles, search parameters, value sets, operation
# definitions, capability statements) are defined by multiple packages with
# the same canonical URL but different version metadata. The IG Publisher
# emits a "multiple potential matches" warning each time the resolver picks
# one. The selection is deterministic (by version pin in sushi-config) and
# the affected references are correct in context; the warning is a known
# multi-version-dependency artifact. Each colliding canonical is enumerated
# explicitly below (no URL-prefix wildcards) per FMG guidance; any newly-
# introduced multi-version collision will surface as an unmatched warning
# for explicit review.

# 6.1.0, 7.0.0) and Da Vinci HRex (1.0.0, 1.1.0, 1.2.0) simultaneously to

# releases of the same OID). The sushi-config parameter

# unversioned canonical references at build time by rewriting them to the
# version in scope for the referencing resource. The information messages

# as configured — expected publisher behaviour, not a defect in the IG.
# Each canonical is enumerated explicitly (no internal wildcards) per FMG
# guidance. Any newly-introduced pin will surface as an unmatched info
